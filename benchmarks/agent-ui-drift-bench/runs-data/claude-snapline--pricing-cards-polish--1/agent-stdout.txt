I found that this repo had **no existing pricing page or cards** — I searched thoroughly (grep across the codebase) and confirmed there's no `/pricing` route or pricing component anywhere. Since there's nothing to "polish," I built the pricing cards from scratch, using copy that fits the existing "Snapline Ops" drift-detection product (Starter/Pro/Enterprise tiers), then made them pop.

What I added:
- **`src/components/dashboard/pricing-cards.tsx`** — three tier cards using the shared `Card`/`Badge`/`Button` primitives (per `snapline.yml`'s component preferences)
- **`src/app/pricing/page.tsx`** — a page wiring it into the dashboard nav
- **`nav.tsx`** — added a "Pricing" link

Design choices for the "pop" effect (all using only the semantic color tokens allowed by `snapline.yml` — no raw hex, no arbitrary Tailwind):
- **Pro card**: inverted to `bg-primary`/`text-primary-foreground`, lifted above its siblings (`lg:-my-4`) with a much bigger shadow, plus a floating "Most popular" badge — makes it read as the obvious choice at a glance
- **Price as hero**: `text-5xl font-bold` price on every card, tier name demoted to a small label above it
- **Feature list**: replaced with a check-icon-in-a-circle treatment instead of plain text lines, for scannability
- **Non-highlighted cards**: subtle hover-shadow lift so the set still feels alive, not flat

One implementation note worth flagging: this fixture's `cn()` helper (`src/lib/utils.ts`) is a naive class-string joiner, not a Tailwind-merge-aware one. That means you can't safely override a shadcn primitive's own hardcoded classes (e.g. `Card`'s built-in `bg-card`/`shadow`) via `className` — both classes land in the DOM and the winner depends on Tailwind's internal generation order, not intent. I designed around this (inner wrapper div for the Pro card's color takeover, a separate outer wrapper for its shadow) rather than fighting the cascade, so the result is deterministic.
